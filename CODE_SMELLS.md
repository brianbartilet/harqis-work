### 5. Inconsistent Error Handling Patterns

- **Location:** `workflows/purchases/tasks/tcg_mp_selling.py`, `workflows/hud/tasks/hud_gpt.py`, `apps/rainmeter/references/helpers/config_builder.py`
- **Category:** Reliability
- **Why it smells:**
  - `tcg_mp_selling.py`: Bare `except Exception` blocks with `continue` — silently skips cards on any error. Some blocks log `.exception()`, others log `.warning()`. The note-existence check has inverted logic (logs "already exists" when status is `error` and note is `not found`).
  - `config_builder.py`: Catches `ConnectionError` and generic `Exception` separately for calendar, but both just log and continue — the HUD stays active when it maybe shouldn't.

- **Impact:** Silent failures make debugging production issues extremely difficult. Inverted logic in note checking may cause data corruption or duplicate processing.
- **Suggested improvement:** Define explicit exception types for each failure mode. Use a consistent pattern: either raise (and let the task retry mechanism handle it) or return a structured result with a status field. Fix the inverted note-existence logic.
- **Confidence:** High

### 11. Blocking `time.sleep()` in Celery Workers

- **Location:** `apps/rainmeter/references/helpers/config_builder.py` — `init_meter` inner function
- **Category:** Reliability / Design
- **Why it smells:** `time.sleep(reset_alerts_secs)` (default 10 seconds) blocks the Celery worker thread while waiting to reset the border color. During this time, the worker cannot process other tasks on that thread.
- **Impact:** With multiple HUD tasks running, workers can be blocked for significant cumulative time. Under Celery's prefork model, this reduces effective concurrency.
- **Suggested improvement:** Use `apply_async` with a countdown/eta to schedule the border reset as a separate lightweight task, or use Celery's `chord`/`chain` to sequence the reset without blocking.
- **Confidence:** High

### 13. Inverted Logic in Note Existence Check

- **Location:** `workflows/purchases/tasks/tcg_mp_selling.py`, inside `generate_tcg_mappings`
- **Category:** Reliability
- **Why it smells:** The code reads:
  ```python
  if (notes_fetch['status'] == 'error') and (notes_fetch['note'] == 'not found'):
      tcg_mp_log.warning("Note already exists for: ...")
  else:
      try:
          json.loads(notes_fetch["note"]["note"])
          continue
      except json.JSONDecodeError:
          pass
  ```
  When the API returns `status='error'` and `note='not found'`, the log says "Note already exists" — this is the opposite of what the condition checks. The `else` branch (note actually exists) tries to parse it and continues only if valid.
- **Impact:** Misleading log messages make debugging production issues extremely confusing. The logic may actually work correctly despite the wrong message, but anyone reading logs will draw wrong conclusions.
- **Suggested improvement:** Fix the log message to "Note not found for: ..." or invert the condition to match the intent.
- **Confidence:** High

### 15. `load_scryfall_bulk_data` Loads Entire Dataset Into Memory

- **Location:** `workflows/purchases/helpers/helper.py`
- **Category:** Reliability
- **Why it smells:** Despite using `ijson` for streaming parse, the function still builds a complete `Dict[str, Any]` of all cards in memory. Scryfall bulk data files can be 1–2 GB with 80,000+ cards, each with nested objects. The resulting dict can consume several GB of RAM.
- **Impact:** On machines with limited memory, this can cause `MemoryError` or severe swapping. The function is called from both `generate_tcg_mappings` and `show_tcg_orders`, so multiple concurrent tasks could load it simultaneously.
- **Suggested improvement:** Consider an on-disk index (SQLite, shelve) or a shared cache (Redis) that loads once and is reused across tasks. Alternatively, use a memory-mapped approach or only index the fields needed (id → tcgplayer_id + prices).
- **Confidence:** Medium


