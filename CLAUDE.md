## Project Overview

**HARQIS-Work** is a Python 3.12+ automation platform built on [harqis-core](https://github.com/brianbartilet/harqis-core). It integrates 16+ third-party applications and runs automated workflows via Celery, n8n, and AI-driven orchestration. Think of it as a code-first RPA platform for business, productivity, and personal systems.

The sole dependency is `harqis-core` (installed from GitHub), which provides all base classes, Celery setup, REST client utilities, Selenium page object base, configuration loaders, and more.

## Commands

```sh
# Install dependencies
pip install -r requirements.txt

# Run all tests (excludes workflows/)
pytest

# Run tests for a specific app
pytest apps/echo_mtg/tests/

# Run a single test file
pytest apps/tcg_mp/tests/test_service.py

# Run only smoke tests
pytest -m smoke

# Run only sanity tests
pytest -m sanity
```

Tests load environment variables from `.env/apps.env` and all app configs from `apps_config.yaml`.

## Architecture

### Apps (`apps/`)

Each app integrates one external service and follows a strict template:

```
apps/<app_name>/
├── config.py                   # Loads this app's section from apps_config.yaml
├── references/
│   ├── base_api_service.py     # Extends harqis-core's BaseFixtureServiceRest
│   ├── base_page.py            # Extends harqis-core's BaseFixturePageObject (Selenium)
│   ├── dto/                    # Dataclass-based data transfer objects
│   ├── models/                 # Data models (often extend DTOs)
│   ├── constants/              # Enums and static values
│   └── web/api/                # Concrete API service implementations
└── tests/
```

`config.py` pattern (same in every app):
```python
APP_NAME = 'ECHO_MTG'
CONFIG = AppConfigWSClient(**load_config[APP_NAME])
```

### Workflows (`workflows/`)

Workflows define Celery tasks that consume app services and push results to the desktop HUD:

```
workflows/<workflow>/
├── tasks_config.py             # Celery Beat schedule (dict of task configs)
├── tasks/                      # @SPROUT.task decorated functions
├── dto/                        # Task parameter DTOs
└── tests/
```

Task decorator chain pattern:
```python
@SPROUT.task(queue='hud')
@log_result()           # Logs output to Elasticsearch
@init_meter(...)        # Initializes Rainmeter desktop widget
@feed()                 # Pushes data to desktop HUD feed
def show_calendar_information(**kwargs):
    ...
```

### Configuration (`apps_config.yaml`)

Central YAML file with a section per app. All sensitive values use `${ENV_VAR}` interpolation. Config is loaded at import time via `ConfigLoaderService` from harqis-core.

### Mapping Files (`workflows.mapping`, `scripts.mapping`)

Auto-generated JSON-like files that expose Celery tasks and shell commands to n8n and AI orchestrators. **Do not edit directly** — they are regenerated by a Celery management job. Keys follow `run-job--<task_name>` or `command-run--<cmd_name>` conventions.

## Key Patterns

**REST API services** extend `BaseFixtureServiceRest` and use a fluent request builder. Responses are deserialized with `@deserialized(dict)` or a specific DTO class.

**Celery tasks** are registered via `@SPROUT.task(queue='<queue>')`. Queues in use: `default`, `hud`, `tcg`. Task schedule configs are dicts in `tasks_config.py` using `timedelta` or `crontab`.

**Tests** use pytest fixtures returning service/page instances, with PyHamcrest assertions (`assert_that`, `equal_to`, etc.).

**New app template**: Copy `workflows/.template/` for a new workflow, or follow the existing `apps/` structure for a new app integration.
